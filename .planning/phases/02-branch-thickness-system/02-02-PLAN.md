  <name>Task 1: Integrate thickness data pipeline and tapered branch rendering into TreeLayer</name>
  <files>src/components/features/background/layers/TreeLayer.tsx</files>
  <action>
Modify TreeLayer.tsx to replace the fixed `widthDecay`/`initialWidth` driven branch rendering with data-driven thickness from cumulative study time. The core changes are inside the `draw` callback:

**1. Import new utilities at top of file:**
- `import { buildCumulativeTimeMap } from '../../../../utils/graphHelpers';`
- `import { timeToThickness, thicknessToColor, thicknessToInactiveColor, MIN_THICKNESS, MAX_THICKNESS } from '../../../../animation/interpolation';`

**2. Add refs for thickness tracking (outside draw callback, alongside existing refs):**
- `lastThicknessUpdateRef = useRef(0)` — timestamp of last thickness spring retarget
- `targetThicknessMapRef = useRef(new Map<string, number>())` — last target thickness per node (to detect changes)
- `knownNodeIdsRef = useRef(new Set<string>())` — track node set for sprout/wither detection
- `witheringNodesRef = useRef(new Map<string, { startX: number; startY: number; endX: number; endY: number; thickness: number; color: string; parentId: string }>())` — cached data for withering branches

**3. Inside draw callback, BEFORE the drawBranch recursion:**

a. Build cumulative time map using `buildCumulativeTimeMap(currentNodes, childrenMap)`. The `childrenMap` is already constructed in the draw callback — reuse it. Note: `buildCumulativeTimeMap` expects `RevisionNode[]` and `Map<string, RevisionNode[]>` — the types already match what TreeLayer has. For the VIRTUAL_ROOT, compute its cumulative time as the sum of all root nodes' cumulative times.

b. Every-minute throttle for thickness spring retargeting:
```
const now = Date.now();
const shouldUpdateThickness = !lastThicknessUpdateRef.current || (now - lastThicknessUpdateRef.current) > 60000;
```
When `shouldUpdateThickness` is true OR when the node set has changed (add/remove detected):
- For each node in `cumulativeMap`, compute `targetThickness = timeToThickness(cumulativeTime)`.
- Compare to `targetThicknessMapRef.current.get(nodeId)`. If difference > 0.3px, retarget the spring: `mgr.set('thick-' + nodeId, currentAnimatedThickness, targetThickness, { stiffness: 60, damping: 14, mass: 1.2 })`.
- For VIRTUAL_ROOT: compute total root cumulative time, apply timeToThickness, use spring key `'thick-VIRTUAL_ROOT'`.
- Update `lastThicknessUpdateRef.current = now`.
- If no spring exists yet for a node (first render), set it immediately: `mgr.set('thick-' + nodeId, targetThickness, targetThickness)` (no animation on initial load).

c. Detect node additions and removals by comparing `currentIds` to `knownNodeIdsRef.current`:
- **New nodes:** Set `mgr.set('ext-' + id, 0, 1, { stiffness: 50, damping: 12, mass: 1 })` for sprout animation.
- **Removed nodes:** Before updating knownNodeIdsRef, cache the last-known rendering data for each removed node into `witheringNodesRef` (startX, startY, endX, endY, thickness, color, parentId — all available from the previous frame's drawBranch). Set `mgr.set('ext-' + id, currentExtension, 0, { stiffness: 70, damping: 16, mass: 0.8 })` for wither animation.
- After processing, update `knownNodeIdsRef.current = currentIds`.

**4. Replace the `drawBranch` function with thickness-aware rendering:**

The `drawBranch` signature changes — instead of receiving a fixed `branchWidth` parameter, it looks up the animated thickness from the spring manager:
```
drawBranch(startX, startY, angle, length, nodeId, depth)
```
(Remove the `branchWidth` parameter.)

Inside drawBranch:
- Look up `animatedThickness = mgr.get('thick-' + nodeId) ?? MIN_THICKNESS`.
- Look up `extensionProgress = mgr.get('ext-' + nodeId) ?? 1` (1 = fully extended). Multiply `length` by `extensionProgress` to implement sprout animation.
- Compute branch color based on thickness: `isActive ? thicknessToColor(animatedThickness) : thicknessToInactiveColor(animatedThickness)`.
- **Taper:** Instead of uniform `ctx.lineWidth` + stroke, draw the branch as a filled tapered polygon:
  - `startWidth = animatedThickness` (thick at base)
  - `endWidth = animatedThickness * 0.55` (taper to ~55% at tip — Claude's discretion for what looks natural)
  - Compute 4 corner points perpendicular to the branch direction, using the start/end widths
  - Use `ctx.beginPath()` -> `ctx.moveTo` -> two `ctx.quadraticCurveTo` calls for smooth edges -> `ctx.closePath()` -> `ctx.fillStyle = color` -> `ctx.fill()`
  - Keep the existing glow/shadow logic but apply to the filled shape (set `ctx.shadowBlur`/`ctx.shadowColor` before `ctx.fill()`)

- **Scale sway inversely with thickness:** `swayAmplitude = 0.02 * (1 + depth * 0.1) * (MIN_THICKNESS / animatedThickness)`. Thick trunk barely sways, thin leaves sway more.
- **Scale glow blur with thickness:** When computing glow for active branches, multiply `style.blur` by `(animatedThickness / MIN_THICKNESS) * 0.5`. This keeps glow proportional to branch width.

- The recursive call to draw children no longer passes `branchWidth * opts.widthDecay`. Children look up their own thickness from the spring manager.

**5. After the main drawBranch recursion, draw withering branches:**
- Iterate `witheringNodesRef.current`. For each withering node, get `extensionProgress = mgr.get('ext-' + id) ?? 0`.
- If `extensionProgress > 0.01`: draw the cached branch data, multiplying length by extensionProgress.
- If `extensionProgress <= 0.01` (animation done): remove from `witheringNodesRef` and `mgr.remove('ext-' + id)`, also `mgr.remove('thick-' + id)`.

**6. Cache branch rendering data each frame for potential wither use:**
- Inside drawBranch, after computing startX/Y, endX/Y, thickness, color: write this data to a `lastBranchDataRef` map (nodeId -> { startX, startY, endX, endY, thickness, color, parentId: ... }). This is read by the remove-detection logic to populate witheringNodesRef.

**7. The `TreeCanvasOptions` interface retains `widthDecay` and `initialWidth` for backward compatibility but they are ignored by the new rendering. Add a comment noting they are deprecated in favor of the data-driven thickness system. Do NOT remove them yet — `useTreeCanvas.ts` (used by RootsBackground) still uses them, and updating roots is Phase 5 scope.**

**Important constraints:**
- Do NOT call Zustand inside the draw callback. All store values are already cached in refs (Zustand-RAF boundary pattern from Phase 1).
- Keep the existing activation wave, breathing glow, and ambient sway logic. Modify sway amplitude scaling and glow blur scaling as described, but do not remove any Phase 1 animation behavior.
- The every-minute throttle applies to thickness spring retargeting only. The animated spring values are still read every frame for smooth rendering.
- Use the existing `springManagerRef.current` for all springs (thickness, extension, activation wave). The spring manager handles many concurrent springs efficiently (verified in research: <5ms for 50 springs).
